# Клиентский контракт формата поля `context`

Единый документ для всех клиентов (**Web UI**, **CLI**), отправляющих `POST /generate` с опциональным полем `context`. Эндпоинт не меняется; задаётся только **содержимое** строки `context`.

Ниже объединены правила прежних версий **v1.1** (англ.) и **v2** (рус.): смысл сохранён, дублирование убрано там, где формулировки совпадают.

---

## Цель

- Снизить **context poisoning** и **prompt injection** через историю.
- Дать **Guard** и основной модели однозначное разделение: диалог vs код.

*(Исходная формулировка v1.1, EN: clients MUST format `context` to prevent Context Poisoning and Prompt Injection; the API shape stays the same.)*

---

## 1. Правило семантической истории

Поле `context` должно содержать **только** семантическую историю диалога и кода.

**Нельзя** включать в `context`:

- системные статусы («Проверка безопасности…», «Попытка 2 из 3…»);
- сырые stack trace и внутренние ошибки API;
- ответы с `return nil -- [SECURITY_BLOCK]...` (не подмешивать блокировку как «нормальный» код в историю).

---

## 2. Смешанный ответ (текст + код из SSE)

Поток SSE отдаёт и `message`, и `code`. В одном ходе ассистента часто есть оба. Клиент **обязан** склеивать их в стандартном виде Markdown, чтобы guard однозначно отделял реплику от кода.

### Шаблон одного хода

````text
User: [ТЕКСТ_ЗАПРОСА]
Assistant: [ТЕКСТ_ИЗ_SSE_message]
```lua
[КОД_ИЗ_SSE_code]
```
````

- Если `message` пустой — строку `Assistant: ...` **не** добавлять.
- Если `code` пустой — блок ` ```lua ` **не** добавлять.

*(Note v1.1: If `message` is empty, omit the Assistant line. If `code` is empty, omit the markdown code block.)*

---

## 3. Пример мульти-туровой строки в JSON `context`

````text
User: Создай массив с 1 и 2
Assistant: Готово.
```lua
local r = _utils.array.new()
table.insert(r, 1)
table.insert(r, 2)
return r
```
User: Замени на 3 и 4
````

---

## 4. Обработка на сервере (невидимо клиенту)

Сервер берёт **уже отформатированный** клиентом `context` и при вызове guard помещает его в изолирующие XML-теги, чтобы снизить риск подмены границ истории:

```xml
<chat_history>
[YOUR STRICTLY FORMATTED CONTEXT STRING]
</chat_history>

<user_input>
[NEW PROMPT HERE]
</user_input>
```

Текущий пользовательский запрос передаётся отдельно в поле `prompt` тела `POST /generate`; строка `context` — только накопленная история в формате выше.

## Приложение A — API Context Formatting Contract (v1.1), английский текст

Ниже — дословное сохранение оригинала **v1.1** для аудита и международных читателей.

To prevent **Context Poisoning** and **Prompt Injection**, all clients (Web UI, CLI) MUST format the `context` field according to these strict rules. The API endpoint (`POST /generate`) remains unchanged, but the *contents* of the `context` string must follow this standard.

### 1. The Rule of Semantic History

The `context` field must ONLY contain purely semantic history.  
It MUST NOT contain:

- System status messages (e.g., "Проверка безопасности...", "Попытка 1 из 3...").
- Internal validation errors or stack traces.
- Security block messages (e.g., `return nil -- [SECURITY_BLOCK]...`).

### 2. Formatting Mixed Responses (Text + Code)

The SSE stream returns both `message` (text) and `code` (Lua). A single assistant turn often contains both. Clients MUST combine them using standard Markdown formatting so the LLM guard correctly parses what was conversational and what was code.

### 3. Example of a Multi-Turn Context String

**What the client sends in the `context` JSON field:**

````text
User: Создай массив и добавь 1 и 2
Assistant: Я создал массив и добавил нужные элементы.
```lua
local r = _utils.array.new()
table.insert(r, 1)
table.insert(r, 2)
return r
```
User: Ой, я имел в виду добавить 3 и 4
Assistant: Понял, исправляю элементы на 3 и 4. Уточните, нужен ли возврат nil при ошибке?
````

### 4. Backend Processing (Invisible to Client)

The backend takes your properly formatted `context` string and securely sandwiches it in XML tags to prevent hijacking:

```xml
<chat_history>
[YOUR STRICTLY FORMATTED CONTEXT STRING]
</chat_history>

<user_input>
[NEW PROMPT HERE]
</user_input>
```
